# Spec 3d — World Surfaces (Town, Who's Who, detail + portraits) — design

**Date:** 2026-06-08
**Status:** Approved (design); plan pending
**Branch (planned):** `feat/world-surfaces-3d`
**Part of:** the "Maple Hollow — World & Reputation" program (Spec 3), the **fourth** of five
sub-specs. **Depends on 3c** (recurring cast & places + generation) being built and merged — 3d is
the presentation layer over 3c's `characters`/`places`/`case_appearances` data.

## Program context (where 3d sits)

3c gives Maple Hollow a persistent, shared cast and places plus per-case appearance tracking, and a
**minimal read-only Who's Who list** as its visible win. 3d turns that data into the **full World**:
a Town screen, a rich Who's Who directory, character and place **detail pages** with cross-case
linking, and the **character/place portrait image pipeline** that 3c deliberately deferred.

**Execution order in the program: 3c → 3d → 3f → 3e.** 3d is the last *purely additive* World layer;
**3f** (open-ended generation + one-tap Start) follows it and depends on a populated world, and **3e**
(the desk home) comes last. 3d itself makes no generation changes.

## The problem 3d solves

After 3c, the world data exists but is barely visible (a flat list, monogram fallbacks, no detail, no
images, no map, no place browsing). 3d makes Maple Hollow a place the kid can explore: see the town,
flip through residents with real portraits, tap a person to read their bio and the cases they turn up
in, and tap a place to see what happened there.

## Decisions (locked with user, 2026-06-08)

- **Portraits land here, not in 3c.** Character portraits (and place images) are generated by a
  pipeline **mirroring the existing avatar/cover image pattern** (gpt-image-1 → local path field,
  non-gating, regenerable). 3c shipped the nullable `portrait_image_path`/`image_path` columns; 3d
  fills them.
- **World surfaces are the full set:** Town (place cards + a light map), full Who's Who directory,
  character detail, place detail — all cross-linked.
- **Spoiler-safe throughout:** every surface obeys 3c's rule — `is_culprit`/`motive` for a case the
  current player hasn't resolved is never shown.

## Goal

Let a Maple Hollow detective **explore the world**: browse the town and its places, page through the
recurring residents with portraits, and follow cross-links between people, places, and the cases they
appear in — all spoiler-safe and in the warm casebook aesthetic.

## Architecture

### Portrait / place-image pipeline (backend)

Mirror the proven TTS/cover/avatar adapter pattern exactly (per `project_stack_facts_actual` and the
casebook image-gen spec):

- A character-portrait generator built on the existing image service (`services/images/`,
  gpt-image-1), with a shared **art-style preamble** (same app-wide style used by covers/avatars) and
  a per-character prompt derived from `name` + `description` + `traits`. Output stored as a local
  path written to `characters.portrait_image_path`; served via the existing static route; cached
  immutable per the project's "audio caching is forever" convention (regeneration writes a new path).
- **Non-gating + regenerable**, exactly like avatars: a synchronous `POST /api/characters/:id/portrait`
  (the avatar pipeline proved a synchronous ~17s image call survives the exe.dev proxy — async not
  needed) and an optional batch path to backfill the seed roster.
- **Spoiler-safe**: portraits are identity-only; no generation prompt may reference a character's
  culprit status in any case.
- Places get the same treatment for `places.image_path` (lower priority; may be a simpler
  `SceneArt`/photo-frame placeholder if image budget is a concern — see open questions).

### World surfaces (frontend — casebook primitives)

All screens reuse the casebook reskin primitives (`Folder`, `Stamp`, `SceneFrame`/`SceneArt`,
index-card paper, typewriter labels) and the inline-`style={{}}` + CSS-vars convention (no CSS
modules), per `project_stack_facts_actual`.

1. **Town screen** — place cards (each a `SceneFrame` with image or `SceneArt` fallback + name +
   short description), plus a **light map**: a simple illustrated/sketch town map with tappable place
   hotspots (v1 = a stylized static map image with positioned labels, **not** an interactive
   pan/zoom engine). Tapping a place → place detail.
2. **Who's Who directory** — the full version of 3c's minimal list: portrait thumbnails (or monogram
   fallback), name, role-flavor blurb, appearance count; tap → character detail. Grouped/sortable
   (e.g. by name; optionally "most-seen").
3. **Character detail** — portrait, bio (description + traits), and **"Appears in these cases"**: a
   spoiler-safe list of `case_appearances` (title + the player's status on that case), each linking
   to that case. Culprit/motive shown **only** for cases this player has resolved.
4. **Place detail** — image, description, and **"Cases set here"** (the mysteries with this
   `place_id`), each linking to the case.

### API additions (3d, extending 3c's reads)

- `GET /api/characters` / `:id` — extend with `portrait_image_path` + place cross-links; keep the
  spoiler redaction from 3c.
- `GET /api/places` / `GET /api/places/:id` — place detail + `cases: [{ mystery_id, title, status }]`
  (spoiler-safe: title/status only; never the solution).
- `POST /api/characters/:id/portrait` (+ optional `POST /api/places/:id/image`) — synchronous
  regenerable image generation.

### Navigation

Entry points to the World are added to the existing home screen as simple links in 3d; the full
**folder-grouped "desk"** reorganization of the home screen is **3e**. (3d makes the surfaces exist
and reachable; 3e makes the home screen *about* them.)

## Components & boundaries

- `services/images/characterPortrait.ts` (+ test) — prompt build + generate + store, mirroring
  `coverImageAgent`/avatar storage. Reuses the shared art-style preamble.
- `routes/characters.ts` / `routes/places.ts` — extended read shapes + the portrait/image POSTs.
- `packages/shared` — extend `Character`/`Place` DTOs with image fields + cross-link shapes.
- `apps/web/src/screens/Town/`, `WhosWho/`, `CharacterDetail/`, `PlaceDetail/` — new screens + routes
  + api client methods. Reuse casebook primitives; no new design tokens (3a/casebook already set
  them).

## Testing

- Portrait generator unit test: ok-path writes a path; provider/storage failure returns ok:false and
  **never throws** (matches `coverImageAgent` fail-soft tests).
- Spoiler-safety tests carried through every new read (character detail + place detail hide unsolved
  culprit/motive).
- Web typecheck + vite build; each screen renders from mocked APIs; map hotspots route to place
  detail; detail pages render the cross-link lists.
- Playwright smoke of Town → place detail → linked case, and Who's Who → character detail → linked
  case.

## Non-goals (deferred)

- **Interactive map engine** (pan/zoom/path-finding) — v1 is a stylized static map with hotspots.
- **HQ folder-desk home** (3e).
- **Editing roster identity** from the UI (names/bios come from seed + generation; only portraits are
  user-regenerable, like avatars).
- **Place images as a hard requirement** — may ship as `SceneArt` placeholders first (open question).
- Any change to the generation pipeline or solvability (that was all 3c).

## Open questions

- **Place images budget:** generate real place images (more gpt-image-1 spend) vs. `SceneArt`/photo-
  frame placeholders for v1. Recommend placeholders first; add real images if the Town feels flat.
- **Seed-roster portrait backfill:** generate all ~12–20 seed portraits up front (one-time image
  cost, user-triggered) vs. lazy on first view. Recommend a user-triggered batch (paid, the user's
  call) + lazy fallback to monogram.
- **Map source:** a single authored town-map illustration (one gpt-image-1 / design asset) vs. a
  composed CSS/SVG sketch. Recommend one authored map image for warmth.

## Acceptance criteria

1. Character portraits generate via gpt-image-1, store a local path, render on Who's Who + detail,
   and are regenerable; failures are fail-soft (monogram fallback, never a crash).
2. Town, Who's Who, character detail, and place detail screens render real 3c data, cross-link
   correctly, and are **spoiler-safe** (no unsolved culprit/motive anywhere).
3. The town map renders with tappable place hotspots routing to place detail.
4. Server suite + web typecheck/build green; deployed to exe.dev for the iPad pass.
