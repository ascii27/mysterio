## Problem

The mystery currently reaches `status: ready` with `narrative_text` populated and `audio_path: null`. Kids read the narrative on screen but never hear it. The original PLAN.md positioned audio as the primary delivery medium (kid listens, takes notes) â€” on-screen text emerged as a secondary affordance during M5b. This milestone closes the gap: turn the narrative into an MP3, store it, serve it, and let the kid press Play on the PlaybackScreen.

## Decisions already locked by the scaffolding

These are not under review â€” they were committed during prior milestones and this design conforms to them:

- **Provider:** OpenAI TTS. `OPENAI_API_KEY` is a required env var; `OPENAI_TTS_MODEL` defaults to `tts-1-hd` and `OPENAI_TTS_VOICE` to `fable` (single narrator, not per-category).
- **Storage:** local filesystem under `AUDIO_DIR` (default `./data/audio`), served via fastify-static at `/audio/*` (registered in `apps/server/src/routes/static.ts`). No S3, no MinIO â€” appropriate for the current scale (one VM at `panther-golem.exe.xyz`).
- **Schema:** `mysteries.audio_path` column exists; `mysteries.status` check constraint already permits `'synthesizing'`.
- **DTO:** `GET /api/mysteries/:id` already returns `audio_url: row.audio_path ? '/audio/${row.audio_path}' : null` (`apps/server/src/routes/mysteries.ts:81`). Frontend `MysteryData` type at `apps/web/src/api/mysteries.ts:13` already includes `audio_url: string | null`.

## Decisions made in brainstorming (2026-05-23)

1. **Scope:** Backend pipeline **+** minimal native `<audio controls>` Play button on PlaybackScreen. Smallest slice that's testable on iPad end-to-end. Custom AudioPlayer (scrubber UI, speed controls, transcript sync) deferred to a later milestone.
2. **TTS failure policy:** 1 inner retry, then soft-fail to `status: ready` with `audio_path: null` and **no** `failure_reason`. Kid still has narrative text + clue tracker; missing audio is silent degradation, not a broken-mystery state.
3. **Code shape:** Provider interface + OpenAI adapter + storage helper + dedicated `ttsAgent` module â€” matches PLAN.md Â§5 verbatim ("treat TTS as a swappable adapter"). Mirrors existing agent pattern (`narrativeAgent`, `validationAgent`).
4. **Audio content:** Only `narrative_text` is sent to TTS. Title stays on-screen only; reading it aloud risks awkward "colon" readings and adds little value when the title is also displayed.
5. **Annotation stripping:** Not needed. `narrative_text` is already stored clean (post-`parseNarrativeAnnotations`); annotation offsets live in `narrative_annotations` JSON separately.
6. **Native HTML5 audio controls.** No custom React audio component. Free play/pause + scrubber + duration + volume + iOS speed-via-long-press; zero new state.

### Provider interface (`services/tts/provider.ts`)

```ts
export interface SynthesizeOpts {
  voice: string;
  model: string;
}

export interface TTSProvider {
  synthesize(text: string, opts: SynthesizeOpts): Promise<Buffer>;
}
```

Buffer (not stream) keeps the contract boring. A 1500-word story is ~150KB of MP3 â€” well below any memory concern. Streaming is a microoptimization we can add if/when it matters.

### OpenAI adapter (`services/tts/openai.ts`)

- Uses the official `openai` npm SDK via `client.audio.speech.create({ model, voice, input, response_format: "mp3" })`.
- Reads model + voice from the `SynthesizeOpts` argument (not directly from env) â€” keeps the adapter testable without env mutation.
- 30s timeout on the call (OpenAI typical p95 for 8K chars is 5â€“10s; 30s gives generous headroom).
- Returns the raw MP3 bytes as a `Buffer`. Surfaces SDK errors as thrown `Error` instances; the agent decides retry policy.

### Storage helper (`services/tts/storage.ts`)

```ts
export async function writeAudio(mysteryId: string, buf: Buffer): Promise<string>;
```

- Writes to `{AUDIO_DIR}/{mysteryId}.mp3`.
- `mkdir -p` the parent dir defensively (`routes/static.ts` already does this at boot â€” duplicate is harmless and protects against test isolation).
- Returns the **relative** path `{mysteryId}.mp3` for storage in `mysteries.audio_path`. The web GET response already prefixes `/audio/`.

### TTS agent (`services/generation/agents/ttsAgent.ts`)

```ts
export async function runTtsAgent(input: {
  mysteryId: string;
  narrativeText: string;
}): Promise<
  | { ok: true; audioPath: string }
  | { ok: false; error: string }
>;
```

Behavior:
1. Read `OPENAI_TTS_MODEL` + `OPENAI_TTS_VOICE` from env (via `loadEnv`).
2. Call `provider.synthesize(narrativeText, {model, voice})`. On error, retry **once** (matches the M4 narrative inner-retry pattern).
3. On success: write the buffer via `storage.writeAudio(mysteryId, buf)`, return `{ok: true, audioPath}`.
4. On exhaustion (both attempts failed): return `{ok: false, error: <last error message>}`.
5. Storage failures do **not** trigger a retry â€” only provider/network errors do. Disk failure is a deployment problem, not a transient API hiccup.

The agent does **not** touch the database; that's the orchestrator's job. Single responsibility.

The agent imports the default provider singleton from `services/tts/index.ts`. Tests substitute the provider via `vi.mock("../tts/index.js", () => ({ defaultProvider: { synthesize: vi.fn() } }))` â€” same pattern used by `narrativeAgent` and `validationAgent` tests for the Anthropic client. The agent does not take a provider parameter on its public signature.

### Audio caching headers

`routes/static.ts` currently uses default fastify-static cache headers. We want the `/audio/` mount to send `Cache-Control: public, max-age=31536000, immutable` (audio for a given `mystery_id` never changes â€” referenced as a project rule in CLAUDE.md). ## Testing strategy

### Unit tests (Vitest, no LLM/network)

1. **`apps/server/src/services/tts/openai.test.ts`** â€” mock the `openai` SDK module via `vi.mock`. Assert:
   - Calls `audio.speech.create({model, voice, input, response_format: "mp3"})` with expected arguments
   - Returns the mocked Buffer bytes unchanged
   - Bubbles up SDK errors as thrown `Error` (no swallowing)

2. **`apps/server/src/services/tts/storage.test.ts`** â€” uses a tmp dir via Vitest's `os.tmpdir()`:
   - Writes a buffer to `{tmpdir}/{id}.mp3` and the file exists with byte-matching content
   - Returns `{id}.mp3` (relative, not absolute)
   - Creates the parent directory if missing

3. **`apps/server/src/services/generation/agents/ttsAgent.test.ts`** â€” `vi.mock` the `services/tts/index.js` module to substitute a fake provider:
   - Success on first call â†’ `{ok: true, audioPath}` after one provider call; storage called once
   - Provider throws on first, succeeds on second â†’ `{ok: true, audioPath}` after two provider calls (retry confirmed)
   - Provider throws twice â†’ `{ok: false, error}` carrying the second error's message; storage never called
   - Provider succeeds, storage throws â†’ `{ok: false, error}`; provider called only once (no retry on storage failure)

### Manual verification on local dev

1. `pnpm dev` (server + web)
2. Generate a fresh mystery (Easy, any category)
3. Watch server logs for: `synthesizing` status, `tts_done {ok: true, audioPath: ...}`, `ready` with `audio_path` set
4. Confirm `data/audio/{mystery_id}.mp3` exists and plays in a media player
5. Open `/mysteries/{id}` â†’ confirm native audio element renders, scrubber populated, tapping Play streams audio
6. Force-failure: temporarily break `OPENAI_API_KEY` (e.g. set to `"invalid"`), regenerate, confirm soft-fail to `ready` with no audio element rendered (kid still has narrative + clue tracker)

### Live LLM test

**None.** Unlike the M2.8 validation pass-rate suite, TTS has no quality-of-output question worth automating. Cost would be $0.24/run on a non-trivial corpus â€” wasteful. Spot-check audio quality manually after deploy.

## Risks & mitigations

- **OpenAI cost.** `tts-1-hd` @ $0.030/1K chars Ã— ~8K chars/mystery = ~$0.24/mystery. Adds ~5Ã— to the per-mystery cost (vs ~$0.05 generation cost today). Not gated this milestone; revisit if generation volume grows beyond family-prototype scale.
- **OpenAI latency.** Adds ~10s to the user-visible "Detective is on the caseâ€¦" wait. The retry on failure can double that to ~20s on a transient error. Acceptable â€” the kid waits ~60s today; another 10s is in the noise.
- **iPad Safari quirks.** Native `<audio controls>` works reliably on iOS Safari with user-gesture-initiated `.play()`. We're not autoplaying. Lock-screen + AirPlay come for free.
- **Disk space.** Each MP3 ~150KB. A thousand mysteries = 150MB. Negligible on the current VM.
- **No CDN.** Single-region FS serving means a kid on a flaky connection across the world streams from `panther-golem.exe.xyz` directly. Acceptable for a family prototype; revisit when there are real users in distant regions.
